# 项目参考资料的目录组织规范

## 概述

在软件开发项目中，参考资料的管理是一项重要的组织工作。本规范定义了项目中 **[[entity-reference-directory|reference 目录]]** 的定位、功能和使用准则，旨在建立一套清晰、可维护的参考资料管理体系。

[[entity-reference-directory|reference 目录]]作为项目中专门划分的只读参考资料存储区域，其核心职能是集中管理各类外部学习材料和技术文档。这些资料包括但不限于：业界优秀的开源项目案例、最新的 SOTA（State of the Art）研究成果对标材料、学术论文、技术博客摘录等。通过统一存放这些参考资料，项目团队能够快速查阅和对比外部资源，同时保持源代码仓库的整洁和专业性。

本规范强调参考资料与源代码的严格分离。这一设计原则基于以下考量：首先，源代码是需要频繁修改和版本控制的核心资产，而参考资料通常是只读的外部引用；其次，参考资料可能涉及许可证合规问题，单独存放便于管理和审查；最后，清晰的目录划分有助于新成员快速理解项目结构，降低学习成本。

## 核心要点

### 基本定位

- [[entity-reference-directory|reference 目录]]仅用于存放只读参考资料，不包含本项目源代码
- 参考资料是外部学习和对标材料的集中存储区
- 目录内容遵循只读原则，不进行主动修改

### 资料类型范围

本目录可包含以下类型的参考资料：

1. **开源项目**：业界优秀的开源项目代码和文档，用于技术对标和学习借鉴
2. **学术论文**：与项目技术方向相关的最新研究成果和学术文献
3. **技术博客**：高质量的技术博客文章、教程和最佳实践总结
4. **SOTA 对标材料**：当前技术领域的最新进展和最佳方案资料

### 大文件处理策略

对于体积较大的参考资料，规范提供了两种推荐方案：

1. **使用 [[concept-git-lfs|Git LFS]]**：对于必须纳入版本控制的二进制文件（如预训练模型、数据集压缩包等），建议使用 Git Large File Storage 进行管理
2. **仅保留外部链接**：对于可通过网络获取的大文件（如公开数据集、第三方库安装包等），建议仅在目录中保留外部链接，避免重复存储

### 目录结构建议

虽然本规范未强制规定目录内部结构，但建议采用以下组织方式以提升可维护性：

```
reference/
├── projects/        # 开源项目代码和文档
├── papers/          # 学术论文和研究报告
├── blogs/           # 技术博客摘录
├── sota/            # SOTA 对标材料
└── externals/       # 外部链接和索引文件
```

## 知识结构

### 目的

在项目仓库中系统化管理外部参考资料，与源代码分离存放，提升仓库整体的可维护性和专业性。

### 方法

设立独立的 [[entity-reference-directory|reference 目录]]作为只读参考资料库，采用分层分类的组织结构，并对大文件采取 LFS 或链接引用的策略。

### 原理

- **关注点分离原则**：将经常变动的源代码与相对稳定的参考资料解耦，降低版本冲突风险
- **空间优化原则**：通过 [[concept-git-lfs|Git LFS]] 和外部链接策略，避免仓库体积过度膨胀
- **知识沉淀原则**：集中管理参考资料便于团队成员查阅学习，促进知识积累和传承

### [[entity-reference-directory|reference 目录]]

作为项目仓库中专门用于存放参考资料的顶级目录，与 src、docs、tests 等目录并列，共同构成完整的项目结构。

### [[concept-git-lfs|Git LFS]]

Git Large File Storage 是 Git 的扩展工具，专门用于管理仓库中的大型二进制文件。在处理 [[entity-reference-directory|reference 目录]]中的大文件时，[[concept-git-lfs|Git LFS]] 能够有效控制仓库体积，同时保留完整的版本控制能力。

### 版本控制系统最佳实践

本规范的设计理念与版本控制的最佳实践一致：通过合理的目录结构和文件管理策略，保持仓库的整洁、高效和可维护性。

## 待探讨

以下问题在本资料中未给出明确答案，需要进一步讨论和确认：

1. **版本同步策略**：当外部参考资料（尤其是开源项目）发生重要更新时，如何及时同步到本地仓库？是否需要建立更新提醒机制？

2. **长期维护机制**：对于存放的参考资料，是否需要定期审查和清理？如何平衡“保留有价值参考”与“避免信息过时”的关系？

3. **链接有效性保障**：对于采用外部链接引用的资料，如何确保链接的长期有效性？是否需要定期检查链接状态？

4. **许可证合规管理**：存放的参考资料可能涉及版权和许可证问题，是否需要建立额外的合规审查流程？

5. **团队协作规范**：多人协作时，如何管理 [[entity-reference-directory|reference 目录]]的修改权限？是否需要设置贡献指南？
